System Map

Subsystems, runtime relationships, and primary data flow.

Primary Data Flow

The main request-response cycle, traced through actual function calls.

  1. 1. User types in InputBar → QtApp._on_send()
  2. 2. QtChannel.on_user_input() → Bus.publish_inbound(InboundMessage)
  3. 3. AgentLoop.run() → bus.consume_inbound() → _process_message(msg)
  4. 4. ContextBuilder.build_cli_turn_prompt() assembles: identity + bootstrap files + memory + skills summary + history
  5. 5a. PTY mode: CLIAgentProvider.send_and_receive() → AgentSession → PTY write → poll snapshots → extract response
  6. 5b. Proxy mode: ProxyAPIProvider.send_and_receive() → HTTP SSE stream → tool-call loop → execute_tool() → yield chunks
  7. 6. Chunks → stream_callback → QtChannel.emit_stream_chunk() → QtApp.receive_assistant_chunk() → ChatPanel (live)
  8. 7. Final response → Bus.publish_outbound(OutboundMessage) → dispatch_outbound() → QtChannel.send()
  9. 8. GUIStore.append_message() persists to JSONL

Side System Flows

Background services that inject messages or update UI.

  • CronService — async timer evaluates due jobs → Bus.publish_inbound() with scheduled message → AgentLoop processes like any user message
  • HeartbeatService — reads HEARTBEAT.md, if non-empty → AgentLoop.process_direct() (bypasses bus, direct turn)
  • UsageMonitor — polls agent CLI for rate-limit data (codex TUI status, claude/gemini startup banner) → on_update callback → QtApp.update_usage() → status bar display
  • Cycle Mode — QTimer in QtApp fires _on_cycle_tick() → publishes continuation message to bus → creates agent-in-loop behavior

Transport Selection

Set in config.json → proxyApi.enabled.
PTY (default): direct terminal session per CLI agent. Best for local dev.
Proxy: HTTP SSE via CLIProxyAPI binary. Enables tool calling, model selection, multi-provider routing. Required for document/email/calendar tools.

Main processing loop. Consumes inbound, builds prompt via ContextBuilder, dispatches to provider, streams response back. Handles loop-mode auto-responder and process_direct() for headless turns.

PTY-based provider. Sends message to AgentSession, polls terminal snapshots for completion, extracts clean response. Special Gemini fallback (non-interactive mode per turn).

Architecture Notes

Key Design Decisions

  • Bus decoupling: MessageBus allows any channel (Qt, TK, future API) to plug in without changing agent core
  • Dual transport: PTY mode for simple local use, Proxy mode for full tool support — selected per-config, not per-session
  • Thread bridge: Qt runs in its own thread; all async→Qt calls go through Signal(object) + _run_on_ui()
  • Streaming: live chunks go via direct stream_callback (bypasses bus for low latency); final response goes via bus
  • Context assembly: progressive — identity first, then bootstrap files, then memory, then skill summaries (full skills loaded on demand)
  • Session persistence: two separate systems — GUIStore (UI cache, JSONL) and SessionManager (agent-loop history). Both active.

Known Tensions / TODOs

  • Dual session stores: GUIStore and SessionManager overlap — conversation lives in both. Needs unification.
  • Time system split: CronService (async timer), HeartbeatService (async sleep), Cycle Mode (QTimer in UI) — three independent timing sources targeting the same bus.
  • No mid-session agent switching: agent is fixed at session creation. Switching requires new session.
  • ExtensionsPanel: partially implemented (Google OAuth works, others stub).
  • Legacy TK GUI: still present, shares channel interface but has diverged in features.
  • Transport per-config not per-session: all sessions use the same provider. Would be useful to mix PTY + Proxy.
